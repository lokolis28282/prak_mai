# ARCHITECT_REVIEW

Дата ревью: 11 июля 2026 года

Целевой релиз: ODE 1.0

Область: backend, HTTP-слой, модульные границы, хранение данных, транзакции, совместимость и эксплуатационная модель.

## 1. Итоговый вывод

ODE сейчас является **модульным монолитом в управляемом переходном состоянии**. Это правильная форма для внутреннего продукта текущего масштаба: один процесс, один локальный файл SQLite, ручная композиция зависимостей и синхронные операции. Переход на микросервисы перед 1.0 не нужен и увеличит эксплуатационный риск.

Архитектура уже имеет полезные границы:

- [`ApplicationContext`](inventory/core/application.py) собирает публичные модули приложения;
- HTTP-слой преимущественно обращается к `WarehouseFacade`, `ReportsFacade` и `AdministrationFacade`;
- новые критичные сценарии склада вынесены в специализированные сервисы и репозитории;
- Reports не пишет в складские таблицы и получает складские факты через контракт чтения событий;
- SQLite-подключения включают внешние ключи и обеспечивают commit/rollback;
- Python API `WarehouseService` и старый способ создания HTTP handler сохранены для совместимости.

Главное ограничение: текущая реализация пригодна только для **одного экземпляра серверного процесса с локальной БД**. Кластер, несколько процессов, сетевой файловый ресурс, автоматический failover и высокая доступность не реализованы и не должны заявляться как поддерживаемые.

До 1.0 не требуется переписывать систему. Требуется зафиксировать поддерживаемую эксплуатационную модель, закрыть оставшиеся опасные обходы фасадов, сделать миграции воспроизводимыми и подтвердить восстановление из резервной копии на копии реальной БД.

## 2. Фактическая схема зависимостей

```text
app.py / CLI / inventory.webapp
                 |
                 v
        ApplicationContext
        /       |        |          \
       v        v        v           v
Warehouse   Reports   Administration  Monitoring
 Facade      Facade       Facade       Facade
   |           |            |            |
   |           |            |            +-- заглушка, без бизнес-таблиц
   |           |            +-- совместимый WarehouseService
   |           +-- Reports services/repository
   |           +-- WarehouseEventReader (только чтение)
   +-- новые Warehouse services/repositories
   +-- совместимый WarehouseService для legacy-сценариев
                 |
                 v
     специализированные adapters
                 |
                 v
          WarehouseCore
                 |
                 v
              SQLite
```

Точка композиции описана в [`inventory/core/application.py`](inventory/core/application.py). Один экземпляр совместимого сервиса передаётся всем фасадам. Это сохраняет единый контекст пользователя и существующее поведение, но одновременно означает, что модульная декомпозиция пока логическая, а не физическая.

Флаги возможностей находятся в [`inventory/core/context.py`](inventory/core/context.py). Warehouse и Reports включены, Monitoring, Mobile и External API отключены. Это соответствует текущему продукту и не создаёт ложного обещания готовности незавершённых модулей.

## 3. Ответственность слоёв

| Слой | Фактическая ответственность | Оценка |
|---|---|---|
| `ApplicationContext` | Создание фасадов, общих контрактов и runtime-конфигурации | Корректная composition root для монолита |
| `inventory/webapp.py` | HTTP, сессии, проверка Origin/Host, маршрутизация, сериализация ответов | Работает, но файл перегружен и содержит legacy-вызовы |
| Facades | Стабильный публичный API модулей, преобразование DB rows в обычные структуры | Правильная целевая граница |
| Domain services | Валидация и оркестрация сценариев прихода, расхода, кабелей, поставок и отчётов | Хорошо выделены в новых сценариях |
| Repositories | SQL и сохранение данных внутри переданного или собственного подключения | Подход соответствует SQLite-монолиту |
| `WarehouseService` | Совместимый Python API и маршрутизация в специализированные adapters | Необходимо сохранить до завершения миграции |
| `WarehouseCore` | Legacy-домен, администрирование, часть SQL и общая блокировка | Главный источник технического долга |
| SQLite | Единое транзакционное хранилище всех модулей | Приемлемо для одного узла |

### ApplicationContext

[`ApplicationContext`](inventory/core/application.py) уже выполняет роль ручного dependency injection и не требует отдельного DI-контейнера. Метод `ensure_application_context()` принимает как новый контекст, так и старый `WarehouseService`; это важная совместимость для тестов, CLI и внешнего Python-кода.

Рекомендация: считать `ApplicationContext` единственной новой точкой входа, но не удалять адаптер совместимости до отдельного мажорного решения.

### Facades

[`WarehouseFacade`](inventory/warehouse/facade.py) содержит как реально мигрированные сервисы, так и делегирование в совместимый сервис. Новые write-path для прихода, расхода, кабелей и приёмки поставки имеют собственные сервисы и репозитории. Это снижает риск постепенно, без массового переписывания.

[`ReportsFacade`](inventory/reports/facade.py) владеет логами работ и импортом отчётов. Складские данные для отчётов приходят через `WarehouseEventReader`, а не через SQL к чужим таблицам.

[`AdministrationFacade`](inventory/administration/facade.py) очищает результаты от секретных полей и предоставляет аудит, пользователей, backup/restore и диагностику. При этом часть административных команд HTTP-слоя всё ещё вызывает `WarehouseService` напрямую.

[`MonitoringFacade`](inventory/monitoring/facade.py) намеренно является изолированной заглушкой и не владеет таблицами. Это соответствует состоянию «В разработке».

### Compatibility layer

[`WarehouseService`](inventory/service.py) является публичным фасадом совместимости. Его специализированные сервисы во многих случаях остаются тонкими adapters над `WarehouseCore`; базовый механизм виден в [`inventory/services/_base.py`](inventory/services/_base.py). Это полезный переходный слой, но не окончательная декомпозиция.

[`WarehouseCore`](inventory/services/warehouse_service.py) остаётся крупным объектом с доменной логикой, SQL, аутентификацией, резервным копированием и общей `RLock`. Удалять его перед 1.0 нельзя: риск регрессий выше выигрыша. Новую логику добавлять туда также не следует, если для неё уже существует фасад модуля.

### Контекст роли и идентичности

Обычная инженерная HTTP-сессия технически использует встроенную запись `lokolis`, но каждый GET и POST входит в `WarehouseService.user_context(..., role_override="engineer")`. Override хранится в `ContextVar`, применяется в `current_user()` и поэтому виден всем сервисным `_require_role()` проверкам текущего request context. В `finally` значение сбрасывается, что исключает сохранение инженерной роли после завершения запроса и не использует общий mutable role field между потоками.

Это закрывает риск повышения прав по принципу defense in depth:

- HTTP-граница требует отдельный session mode для административных операций;
- сервисный слой независимо видит инженера как `role=engineer` и запрещает admin-only методы;
- допустимы только понижающие overrides `engineer`/`viewer`; получить `admin` через override нельзя;
- административный контекст создаётся без override и требует отдельного admin-входа;
- тест [`tests/test_webapp_security.py`](tests/test_webapp_security.py) подтверждает отказ сервиса даже при искусственном обходе route guard.

Остаточный долг относится не к авторизации роли, а к identity: имя инженера в такой сессии является операционной атрибуцией поверх общей технической записи, а не персонально аутентифицированной учётной записью. Для строгой неотказуемости, lifecycle пользователей и централизованного отзыва доступа понадобится персональная identity-модель либо корпоративный SSO/IdP. Это отдельная задача после 1.0 и не должно смешиваться с уже закрытым запретом инженерного доступа к admin-командам.

## 4. Владение данными

Физически все модули используют один SQLite-файл. Владение таблицами является логическим и зафиксировано в [`docs/DATABASE_OWNERSHIP.md`](docs/DATABASE_OWNERSHIP.md).

| Модуль | Таблицы | Комментарий |
|---|---|---|
| Warehouse | `stock_receipts`, `stock_issues`, `stock_issue_allocations`, `deliveries`, `delivery_lines` | Основной актуальный складской контур |
| Warehouse legacy | `equipment`, `operations` | Нужны для совместимости, не являются целевой моделью |
| Reports | `work_logs`, `daily_report_uploads`, `daily_report_rows` | Запись выполняет только Reports |
| Administration | `users`, `audit_log`, backup-файлы | Audit временно используется как источник событий |
| Shared | `reference_values` | Владение смешанное и требует явного решения после 1.0 |
| Monitoring | нет | До реализации модуля таблицы не создаются |

Core не должен владеть бизнес-таблицами. Reports не должен напрямую писать в таблицы Warehouse. Статический аудит этих правил реализован в [`scripts/audit_module_boundaries.py`](scripts/audit_module_boundaries.py) и запускается тестом [`tests/test_architecture.py`](tests/test_architecture.py).

Статический аудит подтверждает отсутствие известных запрещённых импортов и SQL-обходов, но не доказывает независимость модулей во время исполнения. Общий `WarehouseService`, один файл БД и общий контекст пользователя всё ещё связывают модули.

## 5. Ключевые потоки

### Чтение

HTTP GET проходит через `ApplicationContext` в соответствующий фасад. Репозиторий открывает отдельное SQLite-подключение, выполняет запрос, результат преобразуется в обычные `dict/list` и сериализуется HTTP-слоем.

Чтения не удерживают глобальную `service.lock`. `ThreadingHTTPServer` поэтому может обслуживать несколько чтений параллельно в пределах ограничений SQLite и Python-процесса.

### Одиночная запись склада

HTTP POST проходит проверку сессии, входит в контекст пользователя и захватывает `service.lock`. В мигрированных сценариях фасад валидирует данные, сервис вызывает репозиторий, а репозиторий пишет бизнес-запись и аудит в одной транзакции. Для оставшихся legacy-сценариев эту гарантию нужно сохранять контрактными тестами до их переноса.

### Массовый импорт

Preview выполняет разбор и валидацию без изменения БД. Подтверждение повторно получает preview, выполняет пакетную запись через один `connect()` и фиксирует всю операцию целиком. Ошибка внутри пакета приводит к rollback всего контекста подключения.

Preview хранится в памяти процесса, имеет лимиты объёма и TTL, но не переживает рестарт. Это допустимо только при явном UX: после рестарта пользователь должен загрузить файл повторно, а не ожидать продолжения операции.

### Приёмка поставки

[`DeliveryAcceptanceService`](inventory/warehouse/delivery_acceptance.py) использует один transaction context и вызывает контракт receipt repository внутри той же транзакции. Связь строки поставки, создание или дополнение прихода, обновление статуса и audit фиксируются атомарно.

### Отчёты по складу

Reports читает нормализованные события через [`WarehouseEventReader`](inventory/warehouse/events.py). Контракт межмодульных событий описан в [`inventory/core/events.py`](inventory/core/events.py). Текущий `NoopEventPublisher` означает, что система не имеет брокера, durable event bus или асинхронной доставки; операции склада не зависят от подписчиков.

## 6. Транзакции и целостность

Базовый контракт находится в [`inventory/db.py`](inventory/db.py):

- каждое подключение включает `PRAGMA foreign_keys = ON`;
- успешный выход из `connect()` выполняет commit;
- исключение выполняет rollback и повторно выбрасывается;
- подключение всегда закрывается;
- схема содержит PK, FK, CHECK, уникальные и поисковые индексы;
- миграционный код выполняет `foreign_key_check` после перестроения связанной таблицы.

Гарантия атомарности действует в пределах одного SQLite connection. Репозитории, участвующие в общей бизнес-операции, должны принимать уже открытое подключение, как это сделано для приёмки поставки. Открытие второго `connect()` внутри такого сценария создаст отдельную транзакцию и должно считаться архитектурной ошибкой.

Текущая модель не содержит distributed transaction, outbox или двухфазной фиксации. Они не нужны, пока все записи остаются в одной SQLite-БД и нет обязательных внешних подписчиков.

Риски до 1.0:

1. `initialize()` совмещает создание схемы и условные inline-миграции без журнала версий.
2. Нет отдельного идентификатора schema version и проверяемого маршрута обновления для каждой поддерживаемой старой версии.
3. Процессная `RLock` сериализует POST только внутри одного процесса и не защищает от второго процесса.
4. SQLite-конфигурация конкурентного доступа не оформлена как эксплуатационный контракт.
5. `connect()` не задаёт явно journal mode и busy timeout; поведение зависит от настроек файла БД и значений драйвера по умолчанию.

## 7. Поддерживаемая модель развёртывания

### Поддерживается для 1.0

- один экземпляр ODE;
- один Python-процесс;
- `ThreadingHTTPServer` внутри процесса;
- один SQLite-файл на локальном надёжном диске;
- один владелец процесса и каталога данных;
- регулярный SQLite backup и проверяемое восстановление;
- доступ через loopback либо через отдельно управляемый reverse proxy с TLS.

### Не поддерживается

- два и более процесса ODE, работающих с одной БД;
- несколько узлов приложения;
- SQLite-файл на NFS/SMB или другом сетевом ресурсе;
- active-active и active-passive failover;
- балансировщик с несколькими backend instances;
- бесшовное продолжение сессий и preview после перезапуска;
- репликация, автоматическое переключение и zero-downtime deployment.

Причины ограничения:

- сессии и ограничитель входа хранятся в памяти [`inventory/webapp.py`](inventory/webapp.py);
- import previews хранятся в памяти фасадов;
- блокировка записи является локальной `threading.RLock`;
- SQLite-файл не имеет прикладного механизма лидерства или координации между процессами;
- отсутствуют общие session/cache/job stores.

Резервное копирование обеспечивает восстановление после сбоя, но не является HA. Для промышленного запуска это допустимо, если в инструкции явно указаны RPO, RTO, расписание backup, ответственный и проверка восстановления.

## 8. Совместимость API

Совместимость сейчас обеспечивается на двух уровнях:

- Python-клиенты могут продолжать использовать методы и константы `WarehouseService`;
- `make_handler()` принимает как `ApplicationContext`, так и старый `WarehouseService` через `ensure_application_context()`;
- фасады сохраняют старые формы аргументов там, где переход уже начат;
- существующие HTTP paths/actions не требуют смены только из-за модульной декомпозиции.

До 1.0 запрещено удалять `WarehouseService.__getattr__`, старые методы или HTTP actions без инвентаризации потребителей и deprecation-периода. Новые endpoint следует реализовывать через фасады, не расширяя публичную поверхность `WarehouseCore`.

Совместимость не означает, что внутренние private helpers являются долгосрочным контрактом. Их использование CLI и тестами следует отдельно учитывать при последующей миграции.

## 9. Сильные стороны

1. Выбран прагматичный модульный монолит без преждевременного распределения системы.
2. Есть единая composition root и явные публичные фасады.
3. Новые складские write-path имеют специализированные сервисы и репозитории.
4. В мигрированных write-path доменные записи и audit выполняются в одной SQLite-транзакции.
5. Внешние ключи включаются на каждом подключении.
6. Массовые операции имеют общий rollback-контракт.
7. Reports не владеет складскими таблицами и читает их через адаптер событий.
8. Monitoring изолирован до появления реальных требований.
9. Compatibility layer позволяет уменьшать технический долг поэтапно.
10. Границы модулей проверяются автоматизированным статическим аудитом.
11. Инженерская роль принудительно применяется и на HTTP-, и на service-уровне.

## 10. Архитектурные замечания

### P0: блокирующие релиз

Архитектурных дефектов уровня P0, требующих переписывания перед 1.0, не обнаружено. Условие: запуск выполняется строго в поддерживаемой одноузловой модели.

### P1: исправить или формально закрыть до 1.0

1. **Нет формального deployment contract.** Без него второй процесс или размещение БД на сетевой папке могут незаметно нарушить гарантии блокировки и производительности.
2. **Миграции не версионированы.** Условные проверки схемы в `initialize()` сложно воспроизводимо тестировать на нескольких исторических состояниях БД.
3. **HTTP-слой всё ещё обходит фасады.** Legacy-вызовы пользователей, профиля, backup/restore, справочников, старого equipment/import и закрытия поставки идут напрямую через `service`.
4. **Граница Administration неполная.** Фасад существует, но не является единственным путём административных write-операций. Это долг маршрутизации, а не открытый обход ролей: admin-проверка дополнительно выполняется внутри сервиса.
5. **Перезапуск теряет сессии и previews.** Это не порча данных, но должно быть отражено в UX, документации и эксплуатационной инструкции.
6. **Нет подтверждённой матрицы upgrade/restore.** Нужен тест на копии старой БД, аварийное прерывание и восстановление.

### P2: технический долг для 1.1

1. `WarehouseCore` остаётся god object; часть специализированных services является только прокси.
2. `inventory/webapp.py` объединяет HTTP, auth/session, routing, import parsing и часть HTML-представления.
3. `reference_values` имеет смешанное владение Warehouse, Reports и Administration.
4. Legacy-таблицы `equipment`/`operations` сосуществуют с актуальными складскими таблицами.
5. `AuditLogEventReader` является временной проекцией, а `NoopEventPublisher` не даёт durable событий.
6. Полные выгрузки и крупные read-model строятся в памяти; масштабные ограничения разобраны в [`PERFORMANCE_REVIEW.md`](PERFORMANCE_REVIEW.md).
7. Статический boundary audit узок и не проверяет runtime-вызовы, транзакционные границы и обратную совместимость всех сценариев.
8. Инженерские действия имеют человекочитаемого автора, но общая техническая запись не обеспечивает персональную криптографическую идентичность и корпоративный lifecycle доступа.

### P3: только при появлении требований

1. Переход на PostgreSQL.
2. Внешний IdP/SSO.
3. Очередь сообщений и outbox.
4. Несколько процессов или узлов.
5. Репликация, автоматический failover и полноценная HA-топология.

Эти изменения не следует делать «на будущее»: каждое из них меняет эксплуатационную модель и требует отдельного продукта, миграции и QA.

## 11. Решения до 1.0

Обязательные действия:

1. Зафиксировать одноузловой deployment contract в README и runbook: один процесс, локальный диск, путь БД, права, порт/reverse proxy, backup, RPO/RTO и запрет NFS/SMB.
2. Добавить явную schema version и последовательный migration runner либо, минимум для 1.0, реестр применённых миграций и тестируемую матрицу входных версий.
3. Проверить upgrade на обезличенной копии самой старой поддерживаемой production-like БД; до обновления автоматически создавать backup.
4. Проверить restore отдельным прогоном с `integrity_check`, `foreign_key_check` и прикладными smoke-сценариями после восстановления.
5. Перевести административные write-действия HTTP за `AdministrationFacade`; legacy API внутри фасада можно сохранить.
6. Перевести оставшиеся высокорисковые прямые складские write-вызовы HTTP за `WarehouseFacade`, не меняя HTTP contract.
7. Документировать transaction ownership: orchestration service открывает одну транзакцию, вложенные repositories используют переданное connection.
8. Оформить поведение после рестарта: повторный вход, недействительность preview, отсутствие частично подтверждённых операций.
9. Документировать текущую identity-модель: `role_override` обеспечивает авторизацию инженера, а введённое имя является audit attribution, но не персональным логином.
10. Сохранить автоматический architecture-boundary audit и добавить contract-тесты для оставшихся facade/legacy пар.
11. Не запускать два экземпляра приложения на одном SQLite-файле.

## 12. Решения для 1.1

1. Выносить из `WarehouseCore` только законченные вертикальные сценарии с тестами совместимости.
2. Разделить владение `reference_values` или оформить отдельный Reference module с публичным контрактом.
3. Разделить `webapp.py` на router/handler, auth-session, serializers и статическое представление без изменения URL.
4. Ввести версионированный read model и пагинацию для больших списков, поиска и экспорта.
5. Сделать длительные импорты отдельными заданиями с устойчивым состоянием, только если реальные объёмы этого потребуют.
6. Определить срок снятия legacy `equipment/operations` после инвентаризации всех потребителей.
7. Добавить структурированные operational logs, correlation id и метрики запросов/ошибок.
8. Вводить durable outbox только при появлении обязательных внешних интеграций.
9. Спроектировать персональную identity-модель и интеграцию SSO/IdP, если эксплуатационные требования потребуют персональной аутентификации и централизованного отзыва доступа.

## 13. Возможное будущее

Переход к нескольким узлам нельзя выполнить одной настройкой. Он потребует одновременно:

- серверной СУБД вместо общего SQLite-файла;
- общего session store;
- общего preview/job store;
- распределённой координации записей или отказа от process-local lock;
- идемпотентных команд и ключей повторной отправки;
- миграций с совместимостью нескольких версий приложения;
- health/readiness probes, балансировщика и наблюдаемости;
- отдельного плана backup, replication и failover.

До появления подтверждённой потребности это не является задачей ODE 1.0 или 1.1.

## 14. Архитектурные правила

1. `ApplicationContext` является composition root.
2. HTTP обращается к публичным фасадам, а не к `WarehouseCore`.
3. Модуль не пишет в таблицы другого модуля.
4. Межмодульное чтение выполняется через контракт, facade или read model.
5. Одна бизнес-операция использует одну SQLite-транзакцию.
6. Audit записывается в той же транзакции, что и бизнес-изменение.
7. Ошибка в массовой операции откатывает весь пакет, если интерфейс явно не объявляет частичный режим.
8. Новая логика не добавляется в compatibility core без обоснования.
9. Публичная совместимость удаляется только через явный deprecation-процесс.
10. Архитектура развёртывания не считается поддержанной без теста и runbook.
11. Route guard дополняет, но не заменяет проверку роли в сервисном слое.

## 15. Проверка ревью

На момент подготовки документа:

- `python3 scripts/audit_module_boundaries.py` завершается с `module-boundaries: OK`;
- фактические зависимости сверены с `ApplicationContext`, фасадами, services и repositories;
- владение таблицами сверено с `DATABASE_OWNERSHIP.md` и SQL-схемой;
- `role_override` проверен как понижающий request-local контекст с сервисным запретом admin-only методов;
- кластер, HA и multi-process не заявлены как существующие возможности;
- рекомендации не требуют смены схемы БД без обоснованной миграции и не ломают текущий API.

## 16. Честный ответ

С архитектурной точки зрения ODE можно довести до ежедневной эксплуатации без переписывания. Я бы использовал систему каждый день **только после** фиксации одноузлового deployment contract, проверки upgrade/restore на production-like копии и закрытия прямых высокорисковых write-вызовов HTTP за фасадами.

После выполнения пунктов раздела 11 архитектура будет достаточно предсказуемой для ODE 1.0 в заявленной модели: один внутренний сервер, один процесс, локальная SQLite-БД и контролируемое резервное копирование. Для кластера или HA текущая архитектура не готова.
