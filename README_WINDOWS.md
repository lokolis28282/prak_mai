# Запуск ODE на Windows

Документ описывает текущий source/runtime ODE 0.14.0, но последний
фактически собранный Windows ZIP содержит `ODE 0.12.17 RC1`. ZIP для RC2 и
Stage 0.13.2/0.13.3A не создавался. Разделы 9 и 13 относятся только к текущему source
checkout/test contour и отсутствуют в последнем RC1 ZIP. Проверяйте номер
версии внутри конкретного архива до установки; source tree нельзя выдавать за
готовый package.

## 1. Распаковка

Для последнего фактически собранного RC1 распакуйте проверенный ZIP ODE в отдельную папку,
например `C:\ODE`. Не запускайте программу прямо из архива. Для проверки
source Stage 0.13.3A используйте отдельную checkout-папку и только тестовый
launcher; производственный архив из неё в рамках Stage не собирался.

## 2. Запуск

Установите Python 3.10 или новее с сайта python.org и отметьте пункт `Add Python to PATH`. Затем дважды щелкните `start_windows.bat`. Альтернативный запуск из командной строки:

```bat
python app.py
```

Для приемки и экспериментов используйте только `start_test_windows.bat`.
Он заново создаёт `data\warehouse_test_clean.db`, запускает приложение именно
с этой БД и показывает постоянный баннер `ТЕСТОВЫЙ КОНТУР`. Рабочая
`data\warehouse.db` при таком запуске служит read-only источником snapshot.

## 3. Вход

Браузер откроется автоматически. Если этого не произошло, откройте `http://127.0.0.1:8765`. Для рабочей смены укажите ФИО инженера. Для backup, пользователей и диагностики используйте отдельный `Вход администратора`; перенос базы сохраняет пользователей и пароли.

## 4. Где находится база

Рабочая база: `data\warehouse.db`. Не редактируйте и не заменяйте этот файл при запущенной программе.

## 5. Резервная копия перед загрузкой данных

Откройте `Администрирование → Резервные копии` и нажмите `Создать резервную копию`. Дождитесь сообщения об успешном завершении. Копии находятся в `data\backups`.

## 6. Загрузка прихода

Откройте `Склад → Приход`. Для сканирования заполните общие поля партии, установите курсор в поле сканера и считывайте S/N; после проверки списка нажмите `Принять всё на склад`. Ошибочный S/N до подтверждения можно удалить отдельно, выбранной группой или очистить весь временный список. Сканер должен работать как клавиатура и отправлять Enter после кода.

Для CSV скачайте шаблон, выберите заполненный файл, проверьте ошибки и нажмите `Подтвердить загрузку`.

## 7. Загрузка расхода

Для сканирования откройте `Расход`, заполните задачу и общие поля, затем считывайте S/N. Найденные позиции будут показаны до подтверждения; удаление из временного списка не меняет складской баланс. Неизвестные S/N после подтверждения попадут в проблемные строки.

Также можно найти одну позицию и нажать `Списать` либо загрузить CSV массового списания.

## 8. Проверка баланса

Откройте `Склад → Баланс`. Глобальный поиск открывается кнопкой-лупой в компактной шапке и ищет оборудование, поставки и инженеров; поиск баланса работает по всей БД. Пока выполняется серверный поиск, прежние строки скрыты и недоступны для действий. Для полной выборки нажмите `Скачать баланс`.

## 9. Массовое назначение Inventory Number

Сначала сделайте backup согласно разделу 5. Затем откройте
`Склад -> Инвентаризация`, найдите блок `Массовое назначение Inventory Number`
и скачайте отдельный шаблон. В нём обязательны два столбца:

```csv
Serial Number;Inventory Number
SN-000001;INV-000001
```

Сохраните заполненный файл в UTF-8, выберите его и дождитесь Preview. Preview
не меняет БД. Проверьте таблицу и counters:

- `SUCCESS` — номер будет назначен;
- `UNCHANGED` — тот же номер уже записан;
- `NOT_FOUND` — S/N не найден, карточка не создаётся;
- `ALREADY_ASSIGNED` — у позиции уже другой номер;
- `DUPLICATE_INVENTORY_NUMBER` — номер занят другой позицией или повторён в
  CSV;
- `VALIDATION_ERROR` — неверная строка или повтор S/N; Confirm заблокирован.

Если результат верен, нажмите `Подтвердить импорт` и проверьте итоговую
таблицу. Изменяются только строки `SUCCESS`; все они применяются одной
транзакцией. При ошибке выполните новый Preview. Повторный импорт уже
применённого файла должен показать `UNCHANGED`.

Preview и Result показывают максимум первые 100 строк; counters относятся ко
всему файлу. Текущий UI не выводит отдельное предупреждение об усечении.

Этот workflow отличается от расположенной ниже read-only инвентаризации по
одному столбцу S/N: сверка S/N никогда не назначает Inventory Number. Запись
доступна только `engineer/admin`; `viewer` не может подтвердить импорт.

## 10. Поставки

В разделе `Склад → Поставки` загрузите CSV снабжения и проверьте preview: отдельно показаны новые, уже существующие, повторные и ошибочные строки, а также неизвестные столбцы. После подтверждения новые S/N принимаются на склад, а у существующих позиций заполняются только пустые реквизиты. Перед импортом сделайте резервную копию.

## 11. Если CSV открылся одной колонкой

Новые шаблоны и отчеты используют `;` и UTF-8 BOM. Если Excel все равно показал одну колонку, откройте файл через `Данные → Из текста/CSV`, выберите кодировку UTF-8 и разделитель `;`. Один импорт содержит не более 40 000 непустых строк; больший файл разделите до загрузки.

## 12. Завершение работы

Закройте вкладку браузера, затем вернитесь в окно ODE и нажмите `Ctrl+C`. После сообщения `Работа завершена` окно можно закрыть.

## 13. Offline migration staging — Stage 0.13.3A

**IMPLEMENTED:** подготовка справочников и исторических S/N выполняется только
отдельным CLI и не является функцией запущенного ODE. Исходные XLSX/TXT
считаются неизменяемыми, а все candidate-результаты размещаются в ignored
каталоге `migration_inputs\workspace`. Рабочая `data\warehouse.db` открывается
только для read-only проверки и переноса security snapshot в одноразовый
candidate; складские операции из неё не копируются.

```bat
python scripts\migration_reference_data.py inspect-sources
python scripts\migration_reference_data.py build-candidate --overwrite
python scripts\migration_reference_data.py validate-candidate
python scripts\migration_reference_data.py report
```

`report` заново строит allowlisted JSON и применяет тот же path/inode guard;
старый report-файл не читается и не объединяется с новым.

**FACT:** production-схема и плоский `reference_values` этим этапом не
изменяются. Девять таблиц reference/staging создаются только в
`migration_inputs\workspace\warehouse_migration_candidate.db`; candidate DB
не следует запускать как рабочую БД или копировать в `data`.

**FUTURE STAGE:** исторические приходы/расходы и замена рабочей БД выполняются
только отдельным этапом после ручного решения по конфликтам и явного
подтверждения. Stage 0.13.3A ничего не импортирует в production.

## 14. Preservation-aware migration pilot — Stage 0.13.3A.5

**PILOT ONLY / NOT PRODUCTION.** После отдельной сборки pilot artifact можно
открыть только launcher'ом:

```bat
start_migration_pilot_windows.bat
```

Launcher требует уже существующий
`migration_inputs\workspace\warehouse_pilot_candidate.db`, проверяет exact
marker/stage/status, integrity/FK и отсутствие SQLite sidecar, выводит
фактический путь и запускает ODE с баннером `МИГРАЦИОННЫЙ ПИЛОТ`. Он не создаёт,
не очищает и не перезаписывает DB. Не используйте обычный
`start_windows.bat` для pilot DB и никогда не копируйте pilot DB в `data`.
После marker guard обычная инициализация схемы сервиса отключается только для
pilot startup; финальный smoke подтверждает неизменный SHA runtime-копии DB.

Review screen доступен `admin`/`engineer` и показывает 200 selected source rows,
точный S/N, source/canonical names, preservation status, warnings, source
sheet/row и решения. Фильтры разделяют import, quarantine, conflicts и
corrupted. Imported primaries and linked duplicate/conflict history rows may
open the same Equipment Card; non-linked quarantine/corrupted/quantity rows do
not have a card. Все operational writes запрещены backend.

Проверяйте leading-zero/long S/N символ в символ. Numeric и
`SOURCE_CORRUPTED` rows не создают карточки. Полка не является identity. В
фактическом source есть Vegman R220, но нет R200; отсутствующую строку pilot не
генерирует.

Полная инструкция —
[docs/MIGRATION_PILOT_REVIEW_GUIDE.md](docs/MIGRATION_PILOT_REVIEW_GUIDE.md).
Ручное одобрение pilot разрешает только отдельно начать Stage 0.13.3B; оно не
является подтверждением production reset/import.
