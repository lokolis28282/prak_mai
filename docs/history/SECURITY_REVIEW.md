# Security Review ODE, Stage 0.12.17

Дата проверки: 11 июля 2026 года.

## 1. Итог

Текущий контур ODE заметно безопаснее исходной версии: пароли не хранятся в
открытом виде, административные операции проверяются на сервере, добавлены
ограничение попыток входа, срок жизни сессий, защита опасных CSV-значений,
валидация запросов, security headers и проверка внешних ключей SQLite.

При этом продукт пока следует считать приложением для доверенного локального
компьютера. Он **не готов к прямой публикации в корпоративную сеть или интернет**:
в приложении нет TLS termination, SSO, MFA, полноценной CSRF-защиты, централизованного
управления сессиями и защищенного хранилища секретов. Эти возможности не заявлены
как реализованные.

Главные незакрытые риски перед 1.0:

1. новая база по-прежнему создается с известной парой `lokolis/lokolis`;
2. модель Host/Origin и cookie рассчитана на локальный HTTP-контур, а не на
   reverse proxy или многопользовательский сетевой deployment;
3. SQLite-файл и резервные копии не зашифрованы и защищены только правами
   файловой системы.

Вывод: локальное опытно-промышленное использование допустимо после обязательной
смены начального пароля, ограничения доступа к рабочей станции и регулярных
проверяемых backup. Сетевой промышленный запуск до устранения перечисленных
рисков не рекомендуется.

## 2. Область проверки

Проверены:

- вход администратора и вход инженера по ФИО;
- начальный пароль, роли и серверные проверки доступа;
- rate limit и жизненный цикл сессий;
- проверки `Host`, `Origin` и cookie;
- загрузка, проверка и замена production database;
- SQL-запросы и динамически формируемые SQL-фрагменты;
- XSS-границы и CSV formula injection;
- JSON, query parameters, CSV и файловые лимиты;
- HTTP security headers;
- SQLite foreign keys, integrity check, транзакции, rollback и backup;
- focused security tests и существующие contract tests.

Это review исходного кода и автоматических тестов. Оно не заменяет внешний
penetration test, аудит ОС, проверку корпоративного reverse proxy и обследование
прав доступа к рабочим каталогам Windows/macOS.

## 3. Аутентификация

### Реализовано

- Пароли хранятся как `PBKDF2-HMAC-SHA256` с индивидуальной случайной солью и
  260 000 итераций.
- Сравнение хешей выполняется через `hmac.compare_digest`.
- Неактивный пользователь не может пройти password authentication.
- Успешный административный вход записывается в `audit_log`.
- Login принимает только JSON-объект и проверяет типы `mode`, `email`,
  `password`, `full_name`.

### Начальный пароль

При создании пустой базы приложение создает администратора `lokolis` с паролем
`lokolis` и флагом `must_change_password=1`. Серверный guard блокирует
административные операции до смены начального пароля; разрешено только действие
`CHANGE_PASSWORD`. Это предотвращает эксплуатацию admin-функций сразу после
первого входа, но известный пароль все равно остается слабым bootstrap-механизмом.

До 1.0 следует убрать фиксированный пароль: генерировать одноразовый секрет,
получать его из защищенной переменной окружения или требовать создание первого
администратора при инициализации. Секрет не должен попадать в репозиторий,
документацию, лог или release-архив.

### Защита от перебора

Для административного входа действует server-side rate limit:

- ключ: IP-адрес клиента и нормализованный логин;
- окно учета: 5 минут;
- блокировка после 5 неудачных попыток;
- длительность блокировки: 15 минут;
- не более 2 000 записей в памяти процесса;
- успешная аутентификация очищает счетчик.

Во время блокировки API возвращает HTTP `429` и обычный JSON с полем `error`.
Ограничение не является распределенным: перезапуск процесса очищает счетчики,
а смена IP позволяет начать новый ряд попыток. Неудачные входы и блокировки пока
не сохраняются в audit log.

## 4. Сессии

### Реализовано

- После входа создается криптографически случайный opaque token.
- Сессии хранятся только в памяти процесса; перезапуск сервера аннулирует их.
- Cookie имеет атрибуты `HttpOnly`, `SameSite=Strict` и `Path=/`.
- Logout удаляет серверную сессию и cookie.
- Срок неактивности сессии ограничен 12 часами.
- Одновременно хранится не более 500 сессий; при переполнении удаляется самая
  старая созданная сессия.
- Preview поставок дополнительно привязан к session token, поэтому одна смена
  не может подтвердить preview другой смены.

### Ограничения

- Cookie не имеет `Secure`, потому что текущий сервер работает по HTTP.
- Сессии не сохраняются и не синхронизируются между процессами.
- Смена пароля, изменение роли или отключение пользователя не отзывают все уже
  выданные session tokens немедленно.
- Нет административного экрана активных сессий и принудительного logout.
- Нет абсолютного срока жизни отдельно от sliding inactivity timeout.

Эта модель приемлема только для локального процесса. Для сетевого deployment
нужны HTTPS, `Secure` cookie, централизованный session store либо подписанные
короткоживущие токены с механизмом отзыва.

## 5. Роли и административные операции

Сервис различает роли `admin`, `engineer`, `viewer`:

- `viewer` может читать данные, но не выполнять складские write-операции;
- `engineer` может выполнять рабочие приходы, расходы, импорты и отчеты;
- `admin` управляет пользователями, справочниками, backup, integrity check и
  production database.

Проверки выполняются на backend, а не только через скрытие элементов UI.
Административные HTTP-маршруты требуют отдельную admin-session. В частности:

- `/api/admin` и export аудита недоступны обычной смене;
- `/api/upload-prod-db` проверяет admin-session до чтения тела файла;
- сервис замены БД повторно требует роль `admin`;
- focused-тест подтверждает HTTP `403` для engineer-session при загрузке БД.

Обычный вход по ФИО сохраняет совместимость с технической записью `lokolis`, но
web handler передает в service context `role_override="engineer"`. Метод
`current_user()` в этом контексте возвращает роль `engineer`, поэтому сервисные
`_require_role("admin")` остаются второй независимой границей после route guard.
Defense-in-depth тест намеренно отключает HTTP guard и подтверждает, что
`CHECK_DATABASE`/создание backup все равно запрещены. Риск получения admin-прав
обычной сменой через пропущенный route guard считается закрытым.

Переход к индивидуальным учетным записям инженеров остается полезным для более
точной идентификации и будущего SSO, но больше не является blocker текущей
серверной модели полномочий.

## 6. Загрузка production database

Замена рабочей SQLite-базы имеет несколько уровней защиты:

- отдельная admin-session и серверная роль `admin`;
- явный параметр подтверждения;
- расширение `.db` и лимит размера 1 ГБ;
- потоковая запись upload во временный файл;
- `PRAGMA integrity_check`, `PRAGMA foreign_key_check` и проверка обязательных
  таблиц до замены;
- проверка наличия активного администратора после замены;
- safety backup перед операцией;
- атомарная замена файла и rollback к safety backup при ошибке;
- запись результата в audit log.

До 1.0 рекомендуется дополнительно отключать этот маршрут конфигурацией в
обычном рабочем режиме, требовать повторную аутентификацию непосредственно перед
заменой и ограничить максимальный размер фактическими эксплуатационными данными.
Загрузка БД через недоверенную сеть без TLS недопустима.

## 7. SQL injection

В проверенных read/write-сценариях пользовательские значения передаются в
SQLite через placeholders `?`. Поиск с `LIKE`, идентификаторы записей, даты,
фильтры и импортируемые значения не конкатенируются в SQL как данные.

Динамический SQL присутствует для:

- имен таблиц `categories`/`locations`;
- разрешенных полей поставки и карточки;
- заранее заданных filter maps;
- списков placeholders для batch-запросов.

Эти идентификаторы формируются из внутренних allowlist, а не напрямую из HTTP.
Подтвержденного пути SQL injection не найдено.

Остаточный риск связан с поддерживаемостью: централизованного query layer нет,
поэтому каждый новый динамический identifier должен получать явный allowlist и
отрицательный тест с SQL metacharacters. Простого экранирования строки для имени
таблицы или столбца недостаточно.

## 8. XSS и CSV formula injection

### XSS

Основные данные API возвращаются как JSON. В рабочих рендерах пользовательские
значения выводятся через HTML escaping, `textContent` или DOM-компоненты, которые
создают текстовые узлы. Static path отклоняет абсолютные пути и сегменты `..`.
Текст внутренних исключений не возвращается клиенту в ответах HTTP `500`.

Полной гарантии отсутствия stored XSS пока нет: UI содержит много динамических
шаблонов и legacy `innerHTML`, а строгий `Content-Security-Policy` не настроен.
До 1.0 нужны браузерные negative tests со значениями `<script>`, event handlers,
HTML entities и опасными URL во всех импортируемых и редактируемых полях.

### CSV formula injection

Server-side CSV export добавляет апостроф перед строками, которые после пробелов
начинаются с `=`, `+`, `-` или `@`. Client-side CSV для инвентаризации, истории
и ошибок preview использует аналогичную нейтрализацию и CSV quoting. Числовые
отрицательные значения не преобразуются в текстовые формулы.

Focused-тест подтверждает нейтрализацию формулы в server export. При добавлении
нового export необходимо использовать общий helper, а не собирать CSV вручную.

## 9. Валидация входных данных

Реализованы следующие границы:

- JSON action: не более 1 МБ;
- login JSON: не более 100 КБ;
- CSV: не более 50 МБ и 40 000 непустых строк;
- SQLite upload: не более 1 ГБ;
- JSON root обязан быть объектом;
- scalar и collection fields проверяются по ожидаемым типам;
- вложенность JSON collections ограничена;
- целочисленные query parameters имеют контролируемый формат и диапазон;
- обязательные CSV headers нормализуются и проверяются;
- пустые, неизвестные и поврежденные файлы возвращают пользовательскую ошибку;
- повторные S/N, недостаточный остаток и нарушения уникальности проверяются до
  commit;
- generic HTTP `500` не раскрывает traceback, SQL или filesystem paths.

Остаточные риски:

- нет единых максимальных длин для всех строковых полей;
- CSV целиком декодируется в память до обработки;
- 40 000 строк являются контролируемым отказом для файла на 100 000 строк, а не
  поддержкой такого импорта;
- MIME type файла не является доказательством его формата;
- нет антивирусной или content-disarm проверки upload, что особенно важно перед
  будущим сетевым deployment.

## 10. HTTP security headers и CSRF-граница

Обычные HTML, JSON и static responses получают:

- `Cache-Control: no-store`;
- `X-Content-Type-Options: nosniff`;
- `X-Frame-Options: DENY`;
- `Referrer-Policy: no-referrer`;
- `Permissions-Policy` с отключенными camera, microphone и geolocation.

Для POST с заголовком `Origin` сервер требует совпадение `Origin.netloc` и
`Host`, а также разрешенный host. По умолчанию принимаются localhost, loopback и
private IP; дополнительные имена задаются через `ODE_ALLOWED_HOSTS`.
`SameSite=Strict` дополнительно снижает риск browser CSRF.

Это defense in depth, но не полноценная perimeter security:

- запрос без `Origin` не отклоняется;
- отдельного CSRF token нет;
- `Content-Security-Policy` и HSTS отсутствуют;
- trusted proxy headers не определены;
- TLS не реализован встроенным сервером.

Перед любым сетевым запуском необходим документированный reverse proxy profile:
HTTPS, строгий Host allowlist, корректная передача client IP, CSRF token для
state-changing requests и явное решение по CSP/HSTS. До этого сервер следует
оставлять на `127.0.0.1`.

## 11. SQLite, транзакции и целостность

### Реализовано

- Каждое application connection включает `PRAGMA foreign_keys=ON`.
- Context manager выполняет commit только при успехе и rollback при исключении.
- Batch receipts, issues, deliveries и report imports выполняются атомарно.
- Preview не должен оставлять складские записи; issue preview откатывает
  временные изменения.
- Уникальные индексы защищают непустые S/N и инвентарные номера без учета
  регистра.
- Проверка БД включает `integrity_check`, `foreign_key_check` и обязательные
  таблицы.
- Backup создается через SQLite backup API и проверяется перед использованием.
- Restore и production upload создают safety backup и имеют rollback path.

На момент review рабочая `data/warehouse.db` возвращает:

- `PRAGMA integrity_check`: `ok`;
- `PRAGMA foreign_key_check`: ошибок нет.

### Остаточные риски

- База и backup-файлы не зашифрованы at rest.
- Защита от прямого чтения/изменения зависит от прав пользователя ОС.
- Используется `journal_mode=delete`; поведение при длительных конкурентных
  чтениях и записи необходимо учитывать отдельно.
- Audit log не имеет криптографической защиты от изменения владельцем файла.
- Retention и защищенное внешнее хранение backup не автоматизированы.

## 12. Приоритеты до версии 1.0

Обязательные задачи:

1. Удалить фиксированный bootstrap-пароль и реализовать безопасную первичную
   настройку администратора.
2. Зафиксировать поддерживаемую модель deployment. Если нужен доступ 50 инженеров
   по сети, до релиза обязательны TLS/reverse proxy, строгий Host allowlist,
   CSRF-защита, `Secure` cookie и проверка конкурентной нагрузки.
3. Отзывать сессии при смене пароля, роли и отключении пользователя; добавить
   просмотр и принудительное завершение активных сессий.
4. Записывать неудачные admin-login и rate-limit блокировки в security audit без
   сохранения введенного пароля.
5. Добавить строгие максимальные длины полей и browser security fuzz для XSS,
   опасных URL и CSV formulas.
6. Ограничить production DB upload конфигурацией и требовать re-auth перед
   выполнением.
7. Определить права каталогов БД/backup, retention, проверку восстановления и
   хранение копии вне рабочей станции.

## 13. Кандидаты на версию 1.1

- интеграция с корпоративным SSO/AD/OIDC после утверждения инфраструктуры;
- MFA для администраторов;
- более гранулярный RBAC и permissions вместо трех крупных ролей;
- централизованный session/rate-limit store для нескольких процессов;
- интеграция audit events с SIEM и tamper-evident журнал;
- шифрование backup и, при необходимости, БД at rest;
- строгий CSP после удаления оставшихся inline event handlers;
- автоматизированные SAST, dependency scanning и регулярный DAST в CI;
- антивирусная проверка и карантин загружаемых файлов в сетевом контуре.

SSO, MFA, TLS termination, SIEM и сетевой deployment в текущем Stage 0.12.17
не реализованы и не должны описываться пользователям как готовые возможности.

## 14. Проверки

В рамках Stage 0.12.17 подтверждены focused-тестами:

- блокировка admin-login после пяти ошибок и снятие блокировки по TTL;
- session inactivity TTL и лимит количества сессий;
- initial-password guard;
- запрет production DB upload для engineer-session;
- принудительная роль `engineer` в service context и запрет admin-сервисов даже
  при намеренно отключенном HTTP route guard;
- отклонение неверных JSON типов и numeric query parameters без HTTP `500`;
- Host allowlist;
- защита audit export;
- CSV formula neutralization;
- обнаружение поврежденных foreign keys;
- атомарность и rollback основных массовых операций.

Последний полный запуск: 185 тестов, результат `OK`. Focused Stage/security/preview
наборы также прошли. UI smoke после изменений завершен успешно. Сам факт
прохождения тестов не отменяет residual risks и ограничения локальной модели
deployment, описанные выше.
