# Migration ODE 0.12 → 0.13

Статус: **APPROVED PLAN — migration не запускалась**

## Принцип

Side-by-side migration без dual-write. ODE 0.12 замораживается на cutover.
Новая ODE 0.13 DB создается с чистой schema. Legacy data переносится только в
archive/provenance. Первый balance создается будущим approved physical
inventory XLSX.

## Phases

### M0 — freeze manifest

1. объявить maintenance и остановить writes/process;
2. checkpoint/close all handles;
3. выполнить byte-copy закрытого DB и отдельно SQLite Backup API copy;
4. SHA-256/size обоих artifacts;
5. schema dump, PRAGMA report, table/index/trigger counts;
6. row counts и selected aggregate hashes;
7. source file manifest and hashes;
8. application commit/build/config manifest;
9. integrity/FK checks;
10. restore copy в отдельный path и read-only smoke;
11. подписать manifest двумя ответственными.

### M1 — clean candidate

Создать empty 0.13 DB explicit migrations, application_id/schema versions,
initial roles и app_state NOT_INITIALIZED. Никаких старых receipts/issues.

### M2 — users, references, catalog

Перенести identities только по mapping. Known/default credential не переносить.
Approved references переносятся; candidate/pending/rejected попадают в mapping
review/quarantine. В текущей DB все 358 catalog_items_v2 PENDING, поэтому ни
один не становится APPROVED без отдельного решения. До вставки в candidate
parent graphs валидируются на same-domain/same-warehouse и cycle freedom;
parents загружаются до children. Invalid graph блокирует migration, а DB
anti-cycle triggers в обычном runner не отключаются.

### M3 — legacy history

Reconciliation + source registry создают ровно один LegacyHistoryEvent на
source row. Raw payload, serial cells, warnings и hashes сохраняются.
stock_receipts/issues используются для cross-check target link и полезных
полей, но не создают ledger.

### M4 — verification and quarantine

Проверить counts, row hashes, source coordinates, actor/date quality, duplicate
keys, hierarchy invariant queries и отсутствие legacy provenance в
ledger/snapshot/projection. Blocking quarantine не удаляется; acceptance owner
решает ее допустимость.

### M5 — deploy empty operational 0.13

ODE 0.13 стартует с history/search/users/references и NOT_INITIALIZED.
Warehouse posting disabled. Старая DB вне runtime, read-only archive.

### M6 — first physical baseline

Утвержденный FULL XLSX проходит external Preview. Candidate approval создает
Equipment, snapshot и projection at cutoff=0. Только atomic publish переводит
system ACTIVE.

### M7 — acceptance/cutover

Run verification gates, performance/security/restore rehearsal, then controlled
atomic switch. Old binary/config/DB remain rollback set.

### M8 — signed cleanup

После business/data/security acceptance убрать compatibility runtime, archive
one-off tools/docs, удалить tracked release data artifacts и только тогда
обозначить release ODE 0.13.

## No dual-write

Старый и новый balance имеют разные источники истины. Поэтому:

- старое приложение не пишет в новую DB;
- новое приложение не пишет в старую DB;
- нет adapter, копирующего receipt/issue в обе модели;
- между freeze и cutover новые movements запрещены;
- rollback возвращает весь old set, а не отдельные rows.

## Что остается от old DB

data/warehouse.db нельзя удалить:

- immutable archive с manifest/SHA;
- offline migration verification;
- whole-system rollback до окончания window;
- источник расследования расхождений.

Она не монтируется как runtime DB, не входит в balance, package или Git history
нового release. Retention: минимум 10 лет либо longer corporate policy.
