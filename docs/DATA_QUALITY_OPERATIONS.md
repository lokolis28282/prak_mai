# Операции контроля качества данных (ODE 0.15.0)

Экран `Склад → Проблемы` («Контроль качества данных») показывает четыре
категории проблем и с версии 0.15.0 позволяет `engineer`/`admin` исправлять
две из них прямо в таблицах. `viewer` видит все таблицы read-only.

## Категории

| Категория | Источник | Редактирование |
|---|---|---|
| Не сопоставлено | `stock_issues` без полных allocations | только просмотр |
| Дубли S/N | группы `stock_receipts` с одинаковым S/N (`COLLATE NOCASE`) | исправление S/N, удаление лишней карточки |
| Отрицательные остатки | агрегированный баланс < 0 | только просмотр |
| Неполные строки | `stock_receipts` с пустыми shelf/vendor/model | заполнение пустых полей и даты |

«Не сопоставлено» и «Отрицательные остатки» — агрегатные расхождения между
приходами и расходами; их исправление требует сторнирующих операций, которых
в ODE пока нет (см. `TECH_DEBT.md`), поэтому эти таблицы остаются read-only.

## Runtime-путь

Все операции идут только через существующий validated write-путь:

```
UI (static/js/ui.js) → POST /api/action → WarehouseFacade
  → ReceiptWriteService (_require_write: только admin/engineer)
  → ReceiptRepository (транзакция + audit_log)
```

Посадочный guard (`assert_posting_allowed`) блокирует операции в
NOT_INITIALIZED/DEGRADED состояниях склада. Прямой SQL из UI/API запрещён.

## Действия API

### `FILL_RECEIPT_FIELDS`

Payload: `{receipt_id: int, values: {project?, shelf?, vendor?, model?}}`.

Fill-empty-only: записываются только значения в **пустые** поля карточки.
Заполненное поле никогда не перезаписывается; конфликт (поле уже заполнено
другим значением) возвращается в `fill.conflicts` и не применяется. Пустой
ввод игнорируется. Audit: `RECEIPT_FIELDS_FILLED` со списком
`updated_fields`; no-op не создаёт audit-записи.

### `FILL_RECEIPT_DATE`

Payload: `{receipt_id: int, receipt_date: str}`.

Дата прихода — provenance (факт появления позиции на складе), поэтому правило
строже: заполняется **только пустая** дата (пустые даты существуют только у
мигрированных исторических строк — обычный валидатор прихода дату требует).
Формат валидируется (`ГГГГ-ММ-ДД`, `ДД.ММ.ГГГГ`, `ДД/ММ/ГГГГ`), значение
нормализуется в ISO. Заполненная дата никогда не перезаписывается (ответ
`updated: false`). Audit: `RECEIPT_DATE_FILLED` c `manual: true` — дата
проставлена дежурным вручную, а не доказана источником.

### `CORRECT_DUPLICATE_SERIAL`

Payload: `{receipt_id: int, new_serial_number: str}`.

Исправляет S/N одной из карточек группы дублей. Новый S/N: непустой, отличен
от текущего (`casefold`), не занят другой карточкой (`COLLATE NOCASE`,
плюс перехват `IntegrityError` от unique-индекса там, где он есть). Значение
нормализуется `strip().upper()` по общему правилу ввода S/N. Audit:
`RECEIPT_SERIAL_CORRECTED` со старым и новым значением.

### `DELETE_DUPLICATE_RECEIPT`

Payload: `{receipt_id: int}`.

Удаляет **лишнюю дублирующую** карточку. Fail-closed условия (нарушение любого
— отказ с понятным сообщением):

1. карточка существует, S/N непустой;
2. остаётся вторая карточка с тем же S/N — последняя карточка серийника
   удалена быть не может;
3. на карточку не ссылаются `stock_issue_allocations` (списания),
   `delivery_lines` (поставка), `migration_full_identities` /
   `migration_full_reconciliation` (миграционный provenance).

Полный снимок удаляемой строки сохраняется в audit-коде `RECEIPT_DELETED`
(`details.deleted_row`) — операцию можно реконструировать. UI требует
подтверждение перед удалением. Рекомендация: удалять карточку, в которой
меньше данных.

## Timeline

Русские подписи кодов (`static/js/ui.js`, `historyActionLabels`):
`RECEIPT_FIELDS_FILLED` — «Заполнение неполной строки»,
`RECEIPT_DATE_FILLED` — «Ручное проставление даты»,
`RECEIPT_SERIAL_CORRECTED` — «Исправление дубля S/N»,
`RECEIPT_DELETED` — «Удаление дублирующей карточки».

## Тесты

`tests/test_warehouse_data_quality_fix.py` — 12 тестов: fill-empty/конфликты/
no-op audit, валидация и провенанс даты, уникальность и нормализация S/N,
fail-closed удаление (уникальная карточка, наличие списаний), ролевые отказы
`viewer`, полные строки в `duplicate_serials` с сохранением счётчика групп.
Тестовые хелперы воспроизводят legacy-состояние промоутнутой исторической БД
(отсутствие unique-индекса S/N, пустые даты), которое невозможно получить
через обычный write-путь.
